# AI Fitness Platform

A personal, multi-user fitness platform that unifies **Strava** and **WHOOP**
data and layers **Claude** on top for training plans, recovery insights, and
natural-language coaching.

> This is a learning project. The code is heavily commented to explain *why*
> patterns are used, not just what they do.

---

## Stack

| Layer | Tech |
|-------|------|
| Backend | FastAPI, async SQLAlchemy 2.0, asyncpg, Alembic |
| Database | PostgreSQL 16 |
| Auth | JWT (python-jose) + bcrypt (passlib) |
| External APIs | Strava, WHOOP, Anthropic (Claude) via `httpx.AsyncClient` |
| Frontend | Next.js 15 (App Router), React 19, TanStack Query, Tailwind CSS |
| Dev infra | Docker Compose |

## Quick start

### 1. Configure environment

```bash
cp .env.example .env
```

Then edit `.env` and fill in:

- `SECRET_KEY` — a long random string (used to sign JWTs)
- `STRAVA_CLIENT_ID` / `STRAVA_CLIENT_SECRET` — from your Strava API app
- `WHOOP_CLIENT_ID` / `WHOOP_CLIENT_SECRET` — from your WHOOP developer app
- `ANTHROPIC_API_KEY` — from the Anthropic console

The Postgres credentials and `DATABASE_URL` are pre-filled for local Docker use.

### 2. Start the stack

```bash
docker-compose up --build
```

This starts three services:

| Service | URL | Notes |
|---------|-----|-------|
| `db` | `localhost:5432` | Postgres 16, data persisted in the `postgres_data` volume |
| `backend` | `localhost:8000` | FastAPI with hot-reload; Swagger UI at `/docs` |
| `frontend` | `localhost:3000` | Next.js dev server |

### 3. Apply database migrations (required on first run)

The database starts **empty** — nothing auto-creates tables. The initial
migration is committed to the repo, so a fresh checkout just needs:

```bash
docker-compose exec backend alembic upgrade head
```

Later, when you change a model, generate a new migration and apply it:

```bash
docker-compose exec backend alembic revision --autogenerate -m "describe change"
docker-compose exec backend alembic upgrade head
```

> **Why this step exists:** schema changes go through Alembic so dev and
> production stay in sync. Nothing auto-creates tables — that would make it
> impossible to track what's actually deployed. Always read an autogenerated
> migration before applying it; autogenerate can't detect column renames and
> some constraint changes.

Verify everything is up:

```bash
curl http://localhost:8000/health
# {"status":"ok","environment":"development"}
```

## Shutting down

```bash
docker-compose down       # stop containers, keep DB data
docker-compose down -v    # stop AND wipe the database volume (fresh slate)
docker-compose stop       # pause without removing containers
```

## API overview

All routes except `/health`, `/auth/register`, and `/auth/login` require a
`Authorization: Bearer <token>` header. Get a token from `/auth/login`.

| Method | Route | Purpose |
|--------|-------|---------|
| `POST` | `/auth/register` | Create an account |
| `POST` | `/auth/login` | Get a JWT |
| `GET` | `/auth/strava/authorize` | Get the Strava OAuth URL |
| `GET` | `/auth/strava/callback` | OAuth redirect target (stores tokens) |
| `GET` | `/auth/whoop/authorize` | Get the WHOOP OAuth URL |
| `GET` | `/auth/whoop/callback` | OAuth redirect target (stores tokens) |
| `GET` | `/workouts/` | List the current user's workouts |
| `POST` | `/workouts/` | Manually log a workout |
| `GET` | `/workouts/{id}` | Get one workout |
| `DELETE` | `/workouts/{id}` | Delete a workout |
| `POST` | `/workouts/sync` | Pull latest data from Strava + WHOOP |
| `GET` / `POST` | `/recovery/` | Read / write daily recovery records |
| `POST` | `/ai/insights` | Ask Claude a natural-language coaching question |
| `POST` | `/ai/training-plan` | Generate a weekly training plan |

Full interactive docs: **http://localhost:8000/docs**

## Project structure

```
.
├── backend/
│   ├── app/
│   │   ├── routers/     # HTTP layer only — validate + delegate to services
│   │   ├── models/      # SQLAlchemy ORM models (user, workout, recovery, training_plan)
│   │   ├── schemas/     # Pydantic request/response models
│   │   ├── services/    # business logic: strava, whoop, sync, claude
│   │   ├── core/        # config, database engine, security (JWT/hashing)
│   │   ├── dependencies.py  # get_db / get_current_user DI providers
│   │   └── main.py      # app factory, CORS, router registration
│   ├── alembic/         # migration environment + versions/
│   ├── tests/           # pytest + httpx.AsyncClient
│   ├── Dockerfile
│   └── requirements.txt
├── frontend/
│   └── src/
│       ├── app/         # Next.js App Router pages (dashboard, workouts, recovery, ai, auth)
│       ├── api/         # one typed client file per backend router
│       ├── components/  # UI + layout
│       ├── contexts/    # AuthContext
│       └── lib/         # http client, api helpers
├── docker-compose.yml
└── .env.example
```

### Architecture conventions

- **Routers handle HTTP only.** Business logic lives in `services/` so it's
  testable and reusable.
- **Every DB query filters by `user_id`.** This is the multi-user isolation
  boundary — it's not optional.
- **Async everywhere.** Route handlers, services, and external API calls all
  use `async`/`await` so the server isn't blocked on network I/O.
- **Secrets come from `.env`** via `core/config.py` (Pydantic `BaseSettings`).
  Never hardcoded, never committed.

## Local development (without Docker)

### Backend

```bash
cd backend
uvicorn app.main:app --reload        # dev server on :8000
pytest                                # run tests
alembic upgrade head                  # apply migrations
ruff check . && ruff format .         # lint + format
```

### Frontend

```bash
cd frontend
npm install
npm run dev      # dev server on :3000
npm run build    # production build
npm run lint     # ESLint
```
